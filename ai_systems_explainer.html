AI Systems: Detailed Process Guide

LLM · RAG · AI Agent · MCP — with examples, processes, and tools

🧠
Large Language Model (LLM)
The core reasoning engine. Generates text by predicting the most likely next token, trained on billions of text examples. Everything else builds on top of this.
= The Brain — pure intelligence, no persistent memory, no external tools
How it works — step by step
1
Tokenization
Your input text is split into tokens — words, sub-words, or characters. "Hello world" → ["Hello", " world"]. Each token maps to an integer ID in a vocabulary of ~50,000–100,000 entries.
2
Embedding
Each token ID is converted into a high-dimensional float vector (e.g. 4,096 numbers). Similar-meaning words cluster together — "king" and "queen" are mathematically close in this space.
3
Attention layers
Stacked transformer blocks let every token "attend" to every other. In "The cat sat because it was tired", the word "it" learns to strongly attend to "cat" — resolving meaning via context. Repeated across dozens of layers.
4
Next-token prediction
The model outputs a probability distribution over its entire vocabulary. The highest-probability token is selected (or sampled). This repeats autoregressively — one token at a time — until the end token or max length is reached.
5
Detokenization
Token IDs are converted back to human-readable text and returned as the final response. What looks like instant generation is actually thousands of small predictions chained together.

Key limitations
⚠️Knowledge cutoffCannot know events after its training date. No real-time web access by default.
⚠️No memoryEach conversation starts fresh. Cannot remember past sessions without external storage.
⚠️HallucinationMay confidently generate false information, especially on niche or recent topics.
⚠️No actionCan only produce text — cannot send emails, call APIs, or browse the web by itself.
📚
Retrieval-Augmented Generation (RAG)
Adds a live "lookup" step before the LLM responds. Fetches relevant chunks from your knowledge base and injects them as context — grounding answers in real documents.
= Brain + Books — the AI opens a relevant textbook before answering
How it works — step by step
1
Ingest & chunk documents
Your documents (PDFs, wikis, Notion pages, CSVs) are split into ~500-token chunks. Each chunk is a self-contained unit of knowledge that can be retrieved independently.
2
Embed into a vector database
Each chunk is passed through an embedding model (e.g. text-embedding-3-large) to create a vector. These vectors are stored in a vector DB like Pinecone or pgvector. This happens once at setup time.
3
Embed the user's query
When a user asks a question, it's also embedded using the same model — turning it into a vector in the same space as the documents.
4
Semantic search → retrieve top-k chunks
The system finds the most similar chunks via cosine similarity (measuring angle between vectors). Top 3–10 chunks are returned — these are the "most relevant pages from the book."
5
Augment the prompt & generate
Retrieved chunks are injected into the prompt: Context: [chunk1, chunk2...] Question: [user query]. The LLM generates a grounded, citation-able answer using that context.

AI Agent
An LLM given a goal and a set of tools. It autonomously plans, picks tools, observes results, and iterates — completing multi-step tasks without human hand-holding at each step.
= Brain + Hands — can think AND take real-world actions
How it works — the ReAct loop
1
Receive a goal
The agent receives a high-level objective: "Book a meeting with Alice next Tuesday at 2pm and send her an invite." No step-by-step instructions — it must figure out the plan itself.
2
Reason & plan (Thought)
The LLM reasons: "I need to check Alice's calendar, create the event, then email her." This internal chain-of-thought is often invisible to the user but essential to correct execution.
3
Select & call a tool (Action)
It picks the right tool and calls it: calendar.check_availability(user="alice", date="tuesday"). Tools can be: web search, code executor, database, email, file system, APIs.
4
Observe result
The tool returns data. The agent reads it: "Alice is free 2–3pm Tuesday." This observation feeds back into the reasoning loop as new context.
5
Iterate → report
Thought → Action → Observation loops until the goal is reached or the agent determines it's stuck. Finally, it reports the outcome: "Meeting booked and invite sent to Alice."

Model Context Protocol (MCP)
An open standard by Anthropic that lets AI models connect to external services through a unified protocol. Think of it as a universal USB port — one standard that connects the AI brain to any external tool or data source.
= Nervous System — the wiring that connects everything together
How it works — step by step
1
MCP server exposes capabilities
A developer wraps an API (GitHub, Notion, Gmail...) in an MCP server. The server declares: "I offer these tools" (e.g. search_issues, create_pr) and "these resources" (repos, files). Done once, usable by any AI client.
2
MCP client connects & discovers
An AI assistant (Claude, Cursor, etc.) acts as MCP client. At startup it connects to configured MCP servers and discovers their available tools via a handshake — like reading a service's menu.
3
User makes a natural language request
"Show me all open bugs assigned to me." The AI understands this requires an external tool — it can't answer from training data alone.
4
Structured tool call via MCP
The AI sends a structured call: {"tool":"search_issues","args":{"assignee":"me","state":"open"}}. MCP handles transport (HTTP/SSE or stdio), authentication, and serialization invisibly.
5
Result returned & synthesized
The MCP server returns structured data. The AI reads the JSON and synthesizes a natural response: "You have 7 open bugs: [list with priorities]." The user sees clean language, never raw API responses.

Why MCP matters
✅StandardizedOne protocol works with every AI client. Build an MCP server once, connect to Claude, Cursor, or any future AI tool.
✅SecureOAuth-based authentication. The AI never sees your raw credentials — access is scoped and auditable.
✅ComposableAn agent can combine GitHub + Notion + Gmail MCP simultaneously to complete complex cross-tool workflows.
✅Open ecosystemOpen source spec — thousands of MCP servers already exist for popular tools. Anyone can build one.
